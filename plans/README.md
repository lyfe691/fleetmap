## Findings considered and rejected

So nobody re-audits these:

- **"ingest delete+reinsert race shows partial stops to readers"** — false. The
  `ingest_stops` PL/pgSQL function runs in a single transaction; concurrent
  readers see the pre- or post-state, never a partial one. (`0004_orders_stops.sql:128`)
- **"missing index on `stops.vehicle_id`"** — false. The composite
  `(vehicle_id, seq)` index already serves `vehicle_id`-only lookups via its
  leftmost prefix.
- **"`/api/route` needs an explicit dashboard-role check"** — by design. RLS is
  the documented security boundary (`CLAUDE.md` → Conventions); a driver token
  is already blocked by the claim-scoped select policy. A handler-side role check
  would be redundant defense, not a fix.
- **"stale plan docs in `docs/plans/`"** — intentional. Those are per-milestone
  design records kept as history, not living docs.

### Batch 2 (2026-06-22 audit) — considered and rejected / deferred

Rejected (verified false or by-design):

- **Bearer header case-sensitivity** (`startsWith("Bearer ")` after
  `headers.get("authorization")`) — non-issue. WHATWG `Headers.get()` is
  case-insensitive by spec; Next normalizes. Not a finding.
- **Realtime `postgres_changes` vs view RLS coupling** — this is the *documented*
  caveat in `CLAUDE.md` Conventions + the `0004` migration comment; the code is
  correct today. By-design, already recorded.
- **UUID-validate DB-generated order IDs in the ingest RPC** — the IDs are
  Postgres-generated; the user has no control. The audit agent self-concluded
  "no code change needed."

Deferred (real but not worth doing now — don't re-audit without new evidence):

- **Rate-limit `POST /api/location`** — the only remediations need shared state
  (Redis), which the architecture explicitly forbids (stateless API, Supabase
  Realtime fan-out). Threat is a *compromised authed driver* writing to its own
  vehicle row. Revisit only if abuse is observed.
- **Scale-driven map perf** — rail virtualization, RAF-coalescing across vehicle
  markers, snapshot pagination in the live hooks, and aggressive `splitRoute`
  result-caching. All YAGNI for a fleet of tens of vehicles; the map layers are
  already memoized and `splitRoute` already has a forward-clamp cache. Revisit if
  a single dashboard regularly shows ~100+ vehicles.
- **Refactoring the `react-hooks/refs` / `set-state-in-effect` warnings away** —
  these are deliberately downgraded to warnings (see `eslint.config.mjs` and the
  Batch 1 follow-up note above). The "fix" touches animation-critical code; too
  risky before a component test harness exists. Keep as documented posture; if
  the noise bothers anyone, prefer per-line `eslint-disable` with a rationale
  over the risky refactor.
- **Dependency major bumps** (eslint 10, TypeScript 6, `@types/node` 26) — real
  lag, but each is a major with breaking-change risk and no current cost. Bundle
  with a dedicated upgrade pass, not this cleanup. (The trivial, safe
  `next@latest-patch` bump that clears the `postcss` advisory was **not**
  selected into Batch 2 plans; pick it up alongside 008 if desired.)

Not selected into plans this round (audited, available on request): DRY the two
session-mint handlers; handler-hardening cluster (constant-time secret compare,
generic OSRM 404, ingest text-length caps, IDB-error guards in
`use-location-sync`); docs/`.env.example` gaps. All small and low-risk.
